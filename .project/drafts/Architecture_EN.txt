# Architecture Definition Document: Estonian Learning Platform

## 1. Executive Summary

**Goal:** Create a platform for learning Estonian language (LMS Lite), where teachers create interactive lessons and students complete them.
**Key Principles:**

  * **Cost Efficiency:** Scale-to-Zero architecture. Pay only for usage (Pay-per-request).
  * **Maintenance:** "One-man army". Maximum simplicity for support, no servers (Serverless).
  * **User Experience:** "Rich Client" (SPA). Instant interface response.

-----

## 2. High-Level Architecture

The system is built on AWS Serverless. Frontend is completely decoupled from backend.

### Main Components:

1.  **Frontend:** Static SPA (Single Page Application), delivered via CDN.
2.  **Auth:** User and session management via AWS Cognito.
3.  **API Layer:** "Thin" layer of Lambda functions, acting as proxy to database.
4.  **Data Layer:** DynamoDB with separation of Private and Public data.
5.  **Async Worker (Audio):** Isolated circuit for audio generation using heavy ML models.

-----

## 3. Technology Stack

### Frontend

  * **Framework:** React + Vite.
  * **Language:** TypeScript (Strict Mode).
  * **State Management:**
      * *UI/App State:* **Zustand** (for lesson builder and runner).
      * *Server State:* **React Query** (TanStack Query) (for API request caching).
  * **Styling:** Tailwind CSS + Shadcn UI.
  * **Validation:** Zod (shared schemas with backend).

### Backend & Infrastructure

  * **Compute:** AWS Lambda (Node.js) for API.
  * **Compute (Heavy):** AWS Fargate Spot (Docker) for ML models.
  * **Database:** Amazon DynamoDB (On-Demand Capacity).
  * **Queue:** Amazon SQS.
  * **Storage:** Amazon S3 (frontend hosting + media files).
  * **IaC (Infrastructure as Code):**
      * *Foundation (Bases, VPC, Auth):* **Terraform**.
      * *App Logic (Lambdas):* **Serverless Framework** (or SST).

## 4. Data Architecture (Database Design)

**UPDATED:** Uses **one table** DynamoDB with three fields:
- **Partition Key (PK)**
- **Sort Key (SK)**  
- **JSON blob** (all data)

### "Anemic Backend" pattern

Backend contains no business logic. It's just a wrapper over DynamoDB + S3 access.

**Validation:** If frontend sends garbage - it's frontend's problem. Backend will write as is.

Only two types of endpoints:

1. **Public Read API** - reads data marked as public
2. **Private Read/Write API** - reads and modifies private data, automatically filtering by Cognito User ID

### Key Examples:

**Private data (tied to user):**
  * **PK:** `USER#<User_ID>`
  * **SK:** `LESSON#<Draft_ID>`, `PROFILE`, `SETTINGS`

**Public data (access by link):**
  * **PK:** `LESSON#<Lesson_UUID>`
  * **SK:** `METADATA`, `SUBMIT#<Student_ID>`

**Publishing mechanism (SIMPLIFIED):**
~~Copy-on-Write~~ No copying! Draft is created with UUID immediately and stays in place.
- "Publishing" = just sharing the link
- While link is unknown - it's a "draft"
- Theoretically UUID can be guessed, but it's complex

## 5. Audio Subsystem (Async Audio Pipeline)

Implements **"Failover to Generation"** pattern using content-addressable storage (CAS).

### Algorithm:

1.  **Frontend:** Calculates SHA-256 hash of text (e.g., `hash("tere")` -> `abc12345`).
2.  **Read Path:** Tries to load file directly from S3: `GET /audio/cache/abc12345.mp3`.
      * *200 OK:* Play sound. (Cost: ~$0).
      * *404 Not Found:* Send request to API: `POST /generate { text: "tere" }`.
3.  **Write Path (Generation):**
      * API puts task in **SQS**.
      * **AWS Fargate Spot** (container with ML model) spins up on CloudWatch Alarm trigger.
      * Generates mp3, puts in S3 with name `abc12345.mp3`.
      * Frontend periodically polls for readiness (Polling).

**"Pre-warming" Optimization:** When teacher enters Builder, send "empty" message to SQS to warm up Fargate container in advance.

## 6. Code Organization (Frontend Structure)

Application follows "Smart Frontend / Dumb Backend" principle. All business logic is on the client.

```text
src/
├── core/            # PURE LOGIC (Framework Agnostic)
│   ├── types.ts     # Zod schemas and TS interfaces
│   └── logic.ts     # Score calculation functions, hashing
│
├── components/      # "Dumb" UI components (Shadcn)
│
├── features/        # BUSINESS FEATURES (Smart Components)
│   ├── builder/     # Builder logic (Zustand stores)
│   └── runner/      # Lesson completion logic
│
├── services/        # API layer
│   └── api.ts       # Wrapper over fetch
│
└── __tests__/       # Tests
```

## 7. AI-Assisted Development Process

Development is conducted using AI agents (Cursor/Windsurf) following strict protocol.

### Assets:

1.  **Constitution (`.cursorrules`):** Contains stack rules, TDD requirements and references to standards.
2.  **Golden Standard (Reference):** Manually implemented feature-set (e.g., `Flashcard`), containing ideal code, tests and structure.

### Task Assignment Algorithm (Chain of Agents):

1.  **Input:** Human provides **Gherkin (.feature)** file + reference to Golden Standard.
2.  **Step 1 (Architect/Planner):** AI generates implementation plan (file list, JSON changes).
3.  **Step 2 (Critic):** Different AI mode ("Evil Tech Lead") checks plan against Constitution (no Redux, SQL, etc.).
4.  **Step 3 (Coder):** After approval AI writes tests (Red), then code (Green), achieving 80% coverage.

## 8. Security and Limitations

1.  **API:** No Lambda trusts `userId` from request body. Uses only `event.requestContext.authorizer.claims.sub`.
2.  **CORS:** Configured on S3 and API Gateway only for application domain.
3.  **Costs:**
      * DynamoDB: On-Demand.
      * Fargate: Spot Instances + Scale-in cooldown (15 min).
      * Lambda: Minimum memory, short timeouts (except workers).

## 10. Architectural Decision Rationale (Why?)

### Why DynamoDB, not PostgreSQL?
**Answer:** This is a low-load application. Don't want to pay when no users. DynamoDB On-Demand is ideal for scale-to-zero.

### Database: 1 table, not 2!
**Clarification:** Document said 2 tables, but correct is - **one table** with three fields:
- Partition Key (PK)
- Sort Key (SK)
- JSON blob (data)

**"Atomic Backend"** - backend without logic. Only two types of endpoints:
1. **Public Read** - reads data marked as public
2. **Private Read/Write** - reads and modifies private data, automatically filtering by user

### Why Fargate Spot for ML, not Lambda?
**Answer:** Audio generation models are gigantic (~4GB). Very long startup. Paying for startup on each request is expensive. Fargate allows keeping a "warm" container.

### Why React?
**Answer:** Team specializes in React.

### Why one Cognito for everything?
**Answer:** Cognito will be not just for 2 environments, but for **multiple applications** in this AWS account. Goal - **full SSO** (Single Sign-On) between projects.

**Access rights:** No internal roles. Any logged-in user can use application. Login is needed only for:
- So models aren't completely open
- Ability to add rate limiting if needed (doubtful it will be needed)

### Why monorepo?
**Answer:** 
- Don't bother synchronizing front and backend
- Share types and schemas between them

### Why small releases without feature flags?
**Answer:** If feature breaks - rollback entire release to stable version. For a small project this is acceptable.

## 12. Monorepo Structure Analysis

### Option A: Flat Structure (simple)
```
/
├── frontend/          # React application
├── backend/           # Lambdas
├── infra/             # Terraform
└── shared/            # Common types/schemas
```

**Pros:**
- Simple and clear
- Easy to navigate
- Less configuration

**Cons:**
- No dependency isolation
- `npm install` in root can conflict

### Option B: Workspaces (npm/yarn/pnpm)
```
/
├── packages/
│   ├── frontend/      # React application
│   ├── backend/       # Lambdas
│   └── shared/        # Common types (@hak/shared)
└── infra/             # Terraform (outside workspaces)
```

**Pros:**
- Isolated dependencies
- `shared` as internal npm package
- Better scalability

**Cons:**
- More complex to configure
- Need package manager with workspaces

### Decision:

**Option B (Workspaces)** - do it right from the start!

Project is small, but this is a **blueprint for future projects**. Maximum isolation from the beginning.

## 14. Package Manager and TypeScript

### Package Manager: pnpm

**Why not npm/yarn:**
- npm/yarn use hoisting (all dependencies in root node_modules)
- Package can accidentally import undeclared dependency (phantom dependencies)

**Why pnpm:**
- Strict isolation - package sees ONLY its dependencies
- Symlinks instead of copying (saves space)
- If forgot to add dependency - immediate error
- Most popular for serious monorepos (Vercel, Vue, etc.)

### TypeScript Configuration:

**tsconfig structure:**
```
/tsconfig.base.json              # common settings (strict, target, etc)
/packages/frontend/tsconfig.json # extends base + React specifics (jsx, dom)
/packages/backend/tsconfig.json  # extends base + Node specifics
/packages/shared/tsconfig.json   # extends base
```

**Why separate tsconfig:**
- Each package has correct paths and outDir
- Frontend needs DOM types, backend - doesn't
- Common base guarantees consistency

## 16. Build Artifacts Strategy

### Principle: Build Once, Deploy Many

Build is created once, then deployed to different environments.

### Artifact Storage:

**S3 bucket:** `hak-artifacts`
```
s3://hak-artifacts/
  └── builds/
      └── 20241210-143052/    # timestamp for easy cleanup
          ├── index.html
          └── assets/...
```

### CI/CD Flow (GitHub Actions):

1. **Build:** `npm run build`
2. **Upload:** `aws s3 sync dist/ s3://hak-artifacts/builds/${TIMESTAMP}/`
3. **Deploy to dev:** automatically after push
4. **Deploy to prod:** by button (manual trigger)

### Cleanup:

Timestamp in name allows easy deletion of old builds:
- Can set up S3 Lifecycle Policy
- Or script: "delete everything older than 30 days"
