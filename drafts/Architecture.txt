# Architecture Definition Document: Estonian Learning Platform

## 1\. Обзор (Executive Summary)

**Цель:** Создание платформы для изучения эстонского языка (LMS Lite), где учителя создают интерактивные уроки, а студенты их проходят.
**Ключевые принципы:**

  * **Cost Efficiency:** Scale-to-Zero архитектура. Оплата только за использование (Pay-per-request).
  * **Maintenance:** "One-man army". Максимальная простота поддержки, отсутствие серверов (Serverless).
  * **User Experience:** "Rich Client" (SPA). Мгновенный отклик интерфейса.

-----

## 2\. Высокоуровневая Архитектура (High-Level Design)

Система построена на базе AWS Serverless. Фронтенд полностью отделен от бэкенда (Decoupled).

### Основные компоненты:

1.  **Frontend:** Static SPA (Single Page Application), раздается через CDN.
2.  **Auth:** Управление пользователями и сессиями через AWS Cognito.
3.  **API Layer:** "Тонкий" слой Lambda-функций, выступающий прокси к базе данных.
4.  **Data Layer:** DynamoDB с разделением на Приватные и Публичные данные.
5.  **Async Worker (Audio):** Изолированный контур для генерации звука с использованием тяжелых ML-моделей.

-----

## 3\. Технологический Стек

### Frontend

  * **Framework:** React + Vite.
  * **Language:** TypeScript (Strict Mode).
  * **State Management:**
      * *UI/App State:* **Zustand** (для конструктора уроков и раннера).
      * *Server State:* **React Query** (TanStack Query) (для кэширования API запросов).
  * **Styling:** Tailwind CSS + Shadcn UI.
  * **Validation:** Zod (общие схемы с бэкендом).

### Backend & Infrastructure

  * **Compute:** AWS Lambda (Node.js) для API.
  * **Compute (Heavy):** AWS Fargate Spot (Docker) для ML-моделей.
  * **Database:** Amazon DynamoDB (On-Demand Capacity).
  * **Queue:** Amazon SQS.
  * **Storage:** Amazon S3 (хостинг фронта + медиафайлы).
  * **IaC (Infrastructure as Code):**
      * *Foundation (Bases, VPC, Auth):* **Terraform**.
      * *App Logic (Lambdas):* **Serverless Framework** (или SST).

## 4\. Архитектура Данных (Database Design)

**ОБНОВЛЕНО:** Используется **одна таблица** DynamoDB с тремя полями:
- **Partition Key (PK)**
- **Sort Key (SK)**  
- **JSON blob** (все данные)

### "Anemic Backend" паттерн (Вырожденный бэкенд)

Бэкенд не содержит бизнес-логики. Это просто надстройка над DynamoDB + доступ к S3.

**Валидация:** Если фронт отправит мусор - это проблема фронта. Бэкенд запишет как есть.

Только два типа endpoint'ов:

1. **Public Read API** - читает данные, помеченные как публичные
2. **Private Read/Write API** - читает и изменяет приватные данные, автоматически фильтруя по Cognito User ID

### Примеры ключей:

**Приватные данные (привязаны к пользователю):**
  * **PK:** `USER#<User_ID>`
  * **SK:** `LESSON#<Draft_ID>`, `PROFILE`, `SETTINGS`

**Публичные данные (доступ по ссылке):**
  * **PK:** `LESSON#<Lesson_UUID>`
  * **SK:** `METADATA`, `SUBMIT#<Student_ID>`

**Механизм публикации (УПРОЩЕНО):**
~~Copy-on-Write~~ Нет копирования! Черновик создаётся сразу с UUID и остаётся на месте.
- "Публикация" = просто поделиться ссылкой
- Пока ссылка неизвестна - это "черновик"
- Теоретически можно угадать UUID, но он сложный

## 5\. Аудио Подсистема (Async Audio Pipeline)

Реализован паттерн **"Failover to Generation"** с использованием кэширования по содержимому (CAS).

### Алгоритм работы:

1.  **Frontend:** Считает SHA-256 хеш текста (например, `hash("tere")` -\> `abc12345`).
2.  **Read Path:** Пытается загрузить файл напрямую с S3: `GET /audio/cache/abc12345.mp3`.
      * *200 OK:* Играем звук. (Cost: \~$0).
      * *404 Not Found:* Отправляем запрос в API: `POST /generate { text: "tere" }`.
3.  **Write Path (Generation):**
      * API кладет задачу в **SQS**.
      * **AWS Fargate Spot** (контейнер с ML-моделью) поднимается по триггеру CloudWatch Alarm.
      * Генерирует mp3, кладет в S3 с именем `abc12345.mp3`.
      * Фронтенд периодически опрашивает готовность (Polling).

**Оптимизация "Pre-warming":** При входе учителя в Конструктор, отправляется "пустое" сообщение в SQS, чтобы прогреть контейнер Fargate заранее.

## 6\. Организация кода (Frontend Structure)

Приложение следует принципу "Smart Frontend / Dumb Backend". Вся бизнес-логика находится на клиенте.

```text
src/
├── core/            # ЧИСТАЯ ЛОГИКА (Framework Agnostic)
│   ├── types.ts     # Zod схемы и TS интерфейсы
│   └── logic.ts     # Функции подсчета баллов, хеширование
│
├── components/      # "Глупые" UI компоненты (Shadcn)
│
├── features/        # БИЗНЕС-ФИЧИ (Smart Components)
│   ├── builder/     # Логика конструктора (Zustand stores)
│   └── runner/      # Логика прохождения урока
│
├── services/        # Слой API
│   └── api.ts       # Обертка над fetch
│
└── __tests__/       # Тесты
```

## 7\. Процесс Разработки с AI (AI-Assisted Workflow)

Разработка ведется с использованием AI-агентов (Cursor/Windsurf) по строгому протоколу.

### Активы:

1.  **Конституция (`.cursorrules`):** Содержит правила стека, требования TDD и ссылки на эталоны.
2.  **Золотой Эталон (Reference):** Реализованный вручную feature-set (например, `Flashcard`), содержащий идеальный код, тесты и структуру.

### Алгоритм постановки задачи (Chain of Agents):

1.  **Input:** Человек подает **Gherkin (.feature)** файл + ссылку на Золотой Эталон.
2.  **Step 1 (Architect/Planner):** AI генерирует план реализации (список файлов, изменения в JSON).
3.  **Step 2 (Critic):** Другой режим AI ("Злой Tech Lead") проверяет план на соответствие Конституции (нет ли Redux, SQL и т.д.).
4.  **Step 3 (Coder):** После аппрува AI пишет тесты (Red), потом код (Green), добиваясь покрытия 80%.

## 8\. Безопасность и Ограничения

1.  **API:** Ни одна Лямбда не доверяет `userId` из тела запроса. Используется только `event.requestContext.authorizer.claims.sub`.
2.  **CORS:** Настроен на S3 и API Gateway только для домена приложения.
3.  **Costs:**
      * DynamoDB: On-Demand.
      * Fargate: Spot Instances + Scale-in cooldown (15 мин).
      * Lambda: Минимальный объем памяти, короткие таймауты (кроме воркеров).

## 9\. DevOps и Процессы (Q&A с Alex)

### Окружения и Деплой:
- **2 окружения:** dev и prod
- **1 ветка:** `main` (защищена, только через PR)
- **Билд:** один раз, деплой в разные окружения
- **Деплой:** в dev автоматически, в prod по кнопке

### Инфраструктура:
- **Таблицы DynamoDB:** различаются суффиксами (`-dev`, `-prod`)
- **Cognito:** один User Pool, конфигурируется вне проекта
- **Terraform:** управляет DynamoDB, S3
- **Serverless Framework:** управляет API Gateway, Lambdas
- **Terraform state:** S3 + DynamoDB lock (уже есть в конфигурации аккаунта)

### Аудио Кэш:
- Кэш не сбрасывается при деплое
- Хеш = фраза + фонетические настройки + версия модели генератора

### Разработка:
- **Монорепо:** фронт и бэк в одном репозитории
- **Feature Flags:** не нужны (мелкие релизы)
- **Мониторинг:** CloudWatch + Sentry
- **Локальная разработка:** Локальный фронт/бэк идёт напрямую в dev-окружение (DynamoDB, SQS, S3)
- **Версионирование API:** не требуется (единственное приложение, нет публичного API)

### Тестирование:
- **Unit тесты:** Jest везде (фронт и бэк)
- **DynamoDB в тестах:** Не DynamoDB Local! Простой in-memory эмулятор (HashMap: положил/достал по ключу)
- **Бэкенд:** Очень стабильный (2-4 endpoint'a), возможно локальный запуск напротив Lambda dev

### Открытые вопросы:
- ❓ Retry логика при прерывании Fargate Spot во время генерации аудио
- ❓ Rollback стратегия - возможно smoke tests или CloudWatch rollback (посмотрим)
- ❓ Fargate Spot fallback на On-Demand - пока не критично, посмотрим как работает
- ✅ Структура монорепо - Workspaces с полной изоляцией

## 10\. Обоснования архитектурных решений (Почему?)

### Почему DynamoDB, а не PostgreSQL?
**Ответ:** Это мало нагруженная аппликация. Не хотим платить, когда нет пользователей. DynamoDB On-Demand идеально подходит для scale-to-zero.

### База данных: 1 таблица, не 2!
**Уточнение:** В документе было написано 2 таблицы, но правильнее - **одна таблица** с тремя полями:
- Partition Key (PK)
- Sort Key (SK)
- JSON blob (данные)

**"Atomic Backend"** - бэкенд без логики. Только два типа endpoint'ов:
1. **Public Read** - читает данные, помеченные как публичные
2. **Private Read/Write** - читает и изменяет приватные данные, автоматически фильтруя по пользователю

### Почему Fargate Spot для ML, а не Lambda?
**Ответ:** Модели генерации звука гигантские (~4GB). Очень долгий startup. Платить за startup на каждый запрос невыгодно. Fargate позволяет держать "тёплый" контейнер.

### Почему React?
**Ответ:** Команда специализируется на React.

### Почему один Cognito на всё?
**Ответ:** Cognito будет не просто на 2 окружения, а на **множество аппликаций** в этом AWS аккаунте. Цель - **полный SSO** (Single Sign-On) между проектами.

**Права доступа:** Никаких внутренних ролей нет. Любой залогиненный пользователь может использовать приложение. Логин нужен только для:
- Чтобы модели не были полностью открыты
- Возможность добавить rate limiting если понадобится (сомнительно что понадобится)

### Почему монорепо?
**Ответ:** 
- Не заморачиваться синхронизацией фронта и бэкенда
- Шарить типы и схемы между ними

### Почему мелкие релизы без feature flags?
**Ответ:** Если фича сломается - откат всего релиза на стабильную версию. Для маленького проекта это приемлемо.

## 12\. Анализ структуры монорепо

### Вариант A: Плоская структура (простая)
```
/
├── frontend/          # React приложение
├── backend/           # Lambdas
├── infra/             # Terraform
└── shared/            # Общие типы/схемы
```

**Плюсы:**
- Просто и понятно
- Легко навигировать
- Меньше конфигурации

**Минусы:**
- Нет изоляции зависимостей
- `npm install` в корне может конфликтовать

### Вариант B: Workspaces (npm/yarn/pnpm)
```
/
├── packages/
│   ├── frontend/      # React приложение
│   ├── backend/       # Lambdas
│   └── shared/        # Общие типы (@hak/shared)
└── infra/             # Terraform (вне workspaces)
```

**Плюсы:**
- Изолированные зависимости
- `shared` как внутренний npm-пакет
- Лучше масштабируется

**Минусы:**
- Сложнее настроить
- Нужен package manager с workspaces

### Решение:

**Вариант B (Workspaces)** - сразу делаем правильно!

Проект маленький, но это **blueprint для будущих проектов**. Максимальная изоляция с самого начала.

## 13\. CI/CD и Инфраструктура

### CI/CD:
- **GitHub Actions** - не AWS CodePipeline

### Секреты:
- **НИКАКИХ .env файлов!** Вообще в системе их не должно быть
- Все секреты в **AWS Secrets Manager**
- Никакой параметризации через переменные окружения

### Service Discovery:
- Компоненты находят друг друга по **вычисляемым именам**
- Пример: таблица = `hak-lessons-{env}`, bucket = `hak-audio-{env}`

### Домены:
- Базовый домен: **askend-lab.com**
- Примеры:
  - `hak-dev.askend-lab.com` - dev окружение
  - `hak.askend-lab.com` - prod окружение (без "-prod" чтобы не светить другие env)

## 14\. Package Manager и TypeScript

### Package Manager: pnpm

**Почему не npm/yarn:**
- npm/yarn используют hoisting (все зависимости в корневой node_modules)
- Пакет может случайно импортировать необъявленную зависимость (phantom dependencies)

**Почему pnpm:**
- Строгая изоляция - пакет видит ТОЛЬКО свои зависимости
- Symlinks вместо копирования (экономия места)
- Если забыл добавить зависимость - сразу ошибка
- Самый популярный для серьёзных монорепо (Vercel, Vue, и т.д.)

### TypeScript Configuration:

**Структура tsconfig:**
```
/tsconfig.base.json              # общие настройки (strict, target, etc)
/packages/frontend/tsconfig.json # extends base + React специфика (jsx, dom)
/packages/backend/tsconfig.json  # extends base + Node специфика
/packages/shared/tsconfig.json   # extends base
```

**Почему отдельные tsconfig:**
- Каждый package имеет правильные paths и outDir
- Frontend нужны DOM types, backend - нет
- Общий base гарантирует консистентность

## 16\. Build Artifacts Strategy

### Принцип: Build Once, Deploy Many

Билд создаётся один раз, затем деплоится в разные окружения.

### Хранение артефактов:

**S3 bucket:** `hak-artifacts`
```
s3://hak-artifacts/
  └── builds/
      └── 20241210-143052/    # timestamp для удобной очистки
          ├── index.html
          └── assets/...
```

### CI/CD Flow (GitHub Actions):

1. **Build:** `npm run build`
2. **Upload:** `aws s3 sync dist/ s3://hak-artifacts/builds/${TIMESTAMP}/`
3. **Deploy to dev:** автоматически после push
4. **Deploy to prod:** по кнопке (manual trigger)

### Cleanup:

Timestamp в имени позволяет легко удалять старые билды:
- Можно настроить S3 Lifecycle Policy
- Или скрипт: "удалить всё старше 30 дней"
