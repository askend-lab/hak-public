АРХИТЕКТУРА IaC ДЛЯ MONOREPO: ПРИНЦИПЫ И BEST PRACTICES

===================================
CORE PRINCIPLES
===================================

1. SEPARATION OF CONCERNS
Разделяйте инфраструктуру по lifecycle и частоте изменений:
- Shared Infrastructure (меняется редко) - отдельно
- Application Resources (меняются часто) - отдельно
Правило: если ресурс меняется с разной частотой, он должен быть в разных state files

2. BLAST RADIUS CONTROL
Ошибка в одном компоненте не должна влиять на другие:
- Каждый сервис управляет своими ресурсами независимо
- Rollback одного сервиса не затрагивает остальные
- Удаление сервиса не влияет на shared infrastructure

3. RIGHT TOOL FOR THE JOB
Используйте правильный инструмент для каждого типа задач:
- Terraform для infrastructure (то что создается раз и редко меняется)
- Application deployment tools для applications (то что деплоится часто)
Правило: если ресурс деплоится с каждым commit - это не для Terraform

4. STATE ISOLATION
Каждый независимый компонент должен иметь свой state:
- Один state file = один lifecycle
- Параллельные изменения = разные state files
- No shared state между frequently changing components

5. IMMUTABLE INFRASTRUCTURE
То что создано - не изменяется, а пересоздается:
- Lambda functions - новая версия при каждом deploy
- Docker images - новый tag при каждом build
- Shared infrastructure - изменяется через controlled process

===================================
АРХИТЕКТУРА ДЛЯ ВАШЕГО ПРОЕКТА
===================================

LAYER 1: FOUNDATION (Terraform)
Что: Resources которые создаются один раз и редко меняются
Кто управляет: Platform team / DevOps
Частота изменений: Раз в месяц или реже
State: Один terraform.tfstate для всей foundation
Содержит:
- ECR repositories (создаются раз, не удаляются)
- Route53 hosted zones (создаются раз)
- ACM certificates (создаются раз, auto-renew)
- S3 buckets для статики (создаются раз)
- VPC и networking (если нужно, создается раз)
- IAM policies уровня account (если есть shared policies)

Принцип: "Set it and forget it"
Deploy: Только при явных изменениях в infrastructure code
CI/CD: Отдельный workflow или manual approval required

LAYER 2: APPLICATIONS (Serverless Framework или подобные)
Что: Lambda functions, API Gateway, application-specific IAM roles
Кто управляет: Development teams
Частота изменений: Несколько раз в день
State: Каждый сервис имеет свой CloudFormation stack (в случае Serverless)
Содержит:
- Lambda functions
- API Gateway configurations
- Lambda-specific IAM roles
- Custom domain mappings
- Environment variables
- Logs configuration

Принцип: "Deploy often, fail fast, rollback easy"
Deploy: При каждом push в main (если changed files detected)
CI/CD: Automated, parallel, independent

LAYER 3: STATIC CONTENT (Direct S3 upload или CDN)
Что: Frontend builds, static assets
Кто управляет: Frontend team
Частота изменений: Несколько раз в день
State: Stateless (прямая загрузка файлов)
Содержит:
- Built frontend files
- Static assets
- Public content

Принцип: "Just upload it"
Deploy: npm run build && aws s3 sync
CI/CD: Automated, fast, no dependencies

===================================
ВЗАИМОДЕЙСТВИЕ МЕЖДУ LAYERS
===================================

APPROACH: Loose Coupling через Service Discovery

Terraform (Layer 1) экспортирует outputs в SSM Parameter Store:
/infrastructure/{environment}/certificate-arn
/infrastructure/{environment}/hosted-zone-id
/infrastructure/{environment}/ecr-repository-url
/infrastructure/{environment}/domain-name

Serverless (Layer 2) читает эти параметры через ${ssm:/path/to/parameter}

Преимущества:
- Нет прямой зависимости между Terraform и Serverless states
- Можно менять implementation любого layer без влияния на другие
- Service discovery pattern - каждый сервис находит что ему нужно
- Versioning через parameter versions если нужно

===================================
STATE MANAGEMENT STRATEGY
===================================

TERRAFORM STATE:
Location: S3 bucket с versioning enabled
Locking: DynamoDB table для предотвращения concurrent modifications
Structure:
  s3://your-state-bucket/
    infrastructure/
      terraform.tfstate          <- ОДИН state для всей foundation
    
Почему один state для foundation:
- Все shared resources имеют одинаковый lifecycle
- Меняются редко и контролируемо
- Dependencies между ресурсами (certificate -> Route53)
- Проще управлять outputs

SERVERLESS STATE:
Location: CloudFormation stacks (управляется AWS)
Isolation: Каждый сервис = отдельный stack
Naming: {service-name}-{stage}
Examples:
  vabamorf-api-dev
  simplestore-api-dev
  audio-api-dev

Преимущества:
- Automatic state management
- Built-in rollback
- Stack outputs для service discovery
- Parallel deploys без conflicts

===================================
CI/CD PRINCIPLES
===================================

1. CHANGE DETECTION
Deploy только то что изменилось:
- Path filters для определения affected services
- Skip unchanged components
- Parallel execution где возможно

Implementation:
  uses: dorny/paths-filter@v2
  filters:
    infrastructure: 'infrastructure/terraform/**'
    vabamorf: 'services/vabamorf-api/**'

2. DEPENDENCY ORDERING
Правильная последовательность:
  Foundation (Terraform) -> Applications (Serverless) -> Static Content

Но: Applications могут деплоиться параллельно между собой

Implementation через needs:
  deploy-services:
    needs: [deploy-infrastructure]

3. ENVIRONMENT PROMOTION
Dev -> Staging -> Production через одинаковый процесс:
- Same code
- Same tools
- Same commands
- Different variables

Implementation:
  STAGE=${{ github.ref == 'refs/heads/main' && 'prod' || 'dev' }}

4. FAIL FAST, ROLLBACK EASY
- Каждый service может откатиться независимо
- Infrastructure changes требуют approval
- Automatic rollback при failures

===================================
DOCKER IMAGES STRATEGY
===================================

BUILD ONCE, DEPLOY MANY:
1. Build Docker image в CI/CD
2. Tag с git commit SHA
3. Push в ECR
4. Serverless использует этот image

Serverless автоматически:
- Видит новый image в ECR
- Создает новую Lambda version
- Обновляет $LATEST alias
- Сохраняет предыдущие versions для rollback

Lifecycle:
- Terraform создает ECR repository (раз)
- Terraform настраивает lifecycle policy (сохранять последние N images)
- CI/CD пушит images при каждом deploy
- Serverless использует latest image

===================================
SCALING PATTERN
===================================

ADDING NEW SERVICE:
1. Проверить нужны ли новые foundation resources (обычно нет)
2. Создать директорию services/new-service/
3. Добавить serverless.yml
4. Добавить path filter в CI/CD
5. Deploy

Не требуется:
- Изменять Terraform
- Трогать другие сервисы
- Координировать deploys

===================================
TROUBLESHOOTING PRINCIPLES
===================================

STATE LOCK ISSUES:
Root cause: Multiple processes trying to modify same Terraform state
Solution: Разделить state по lifecycle boundaries

CIRCULAR DEPENDENCIES:
Root cause: Tight coupling между layers
Solution: Service discovery через SSM parameters

SLOW DEPLOYS:
Root cause: Sequential processing когда можно parallel
Solution: Change detection + parallel execution

ROLLBACK DIFFICULTIES:
Root cause: Monolithic state с множеством resources
Solution: Independent stacks per service

===================================
MIGRATION STRATEGY
===================================

PHASES:

Phase 1: Create new structure (no changes to running infrastructure)
- Создать новую directory structure
- Написать новые Terraform configs (только shared resources)
- Написать Serverless configs для каждого service
- НЕ применять изменения

Phase 2: Parallel run (both old and new exist)
- Deploy new infrastructure alongside old
- Использовать разные имена ресурсов
- Проверить что работает
- Переключить traffic постепенно

Phase 3: Migrate state (careful!)
- Remove resources from old Terraform state
- Import into new structure
- Verify no changes in terraform plan

Phase 4: Cleanup old resources
- Удалить старые ресурсы
- Удалить старый code
- Update documentation

Phase 5: Optimize
- Fine-tune CI/CD
- Adjust timeouts и retries
- Monitor и improve

===================================
KEY DECISIONS SUMMARY
===================================

DECISION: Terraform for infrastructure, Serverless for applications
REASON: Different lifecycles require different tools
ALTERNATIVE REJECTED: All-Terraform approach
WHY REJECTED: Would require complex state splitting, slower deploys, worse rollback story

DECISION: SSM Parameter Store for service discovery
REASON: Loose coupling between layers, no direct state dependencies
ALTERNATIVE REJECTED: Terraform outputs consumed directly
WHY REJECTED: Creates tight coupling, harder to evolve independently

DECISION: Change detection in CI/CD
REASON: Faster deploys, only deploy what changed
ALTERNATIVE REJECTED: Deploy everything always
WHY REJECTED: Wastes time, higher risk of breaking working services

DECISION: Parallel service deploys
REASON: Independent services should deploy independently
ALTERNATIVE REJECTED: Sequential deploys
WHY REJECTED: Slow, unnecessary serialization

===================================
SUCCESS METRICS
===================================

You know the architecture is correct when:
- Can deploy one service without touching others
- State lock conflicts исчезли
- Deploy time пропорционален размеру изменений (not fixed overhead)
- Rollback одного сервиса не требует rollback других
- Adding new service требует минимальных изменений в infrastructure
- Developers могут deploy без DevOps помощи
- Infrastructure changes происходят редко и контролируемо

===================================
ANTI-PATTERNS TO AVOID
===================================

DON'T: Put frequently changing resources in same Terraform state as rarely changing
DON'T: Use Terraform for resources that deploy with every commit
DON'T: Share state between independent services
DON'T: Deploy everything when only one thing changed
DON'T: Use complex dependencies between Terraform modules when simple service discovery would work
DON'T: Put application configuration in Terraform (use application config management)
DON'T: Manual state file manipulation without backups
