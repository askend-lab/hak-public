IaC ARCHITECTURE FOR MONOREPO: PRINCIPLES AND BEST PRACTICES

===================================
CORE PRINCIPLES
===================================

1. SEPARATION OF CONCERNS
Separate infrastructure by lifecycle and frequency of changes:
- Shared Infrastructure (changes rarely) - separate
- Application Resources (changes frequently) - separate
Rule: if resource changes at different frequency, it should be in different state files

2. BLAST RADIUS CONTROL
Error in one component should not affect others:
- Each service manages its resources independently
- Rollback of one service doesn't affect others
- Service deletion doesn't affect shared infrastructure

3. RIGHT TOOL FOR THE JOB
Use the right tool for each type of task:
- Terraform for infrastructure (what's created once and rarely changes)
- Application deployment tools for applications (what's deployed frequently)
Rule: if resource deploys with each commit - it's not for Terraform

4. STATE ISOLATION
Each independent component should have its own state:
- One state file = one lifecycle
- Parallel changes = different state files
- No shared state between frequently changing components

5. IMMUTABLE INFRASTRUCTURE
What's created - doesn't change, but is recreated:
- Lambda functions - new version with each deploy
- Docker images - new tag with each build
- Shared infrastructure - changes through controlled process

===================================
ARCHITECTURE FOR YOUR PROJECT
===================================

LAYER 1: FOUNDATION (Terraform)
What: Resources created once and rarely changed
Who manages: Platform team / DevOps
Frequency of changes: Once a month or less
State: One terraform.tfstate for entire foundation
Contains:
- ECR repositories (created once, not deleted)
- Route53 hosted zones (created once)
- ACM certificates (created once, auto-renew)
- S3 buckets for static content (created once)
- VPC and networking (if needed, created once)
- IAM policies at account level (if there are shared policies)

Principle: "Set it and forget it"
Deploy: Only with explicit changes in infrastructure code
CI/CD: Separate workflow or manual approval required

LAYER 2: APPLICATIONS (Serverless Framework or similar)
What: Lambda functions, API Gateway, application-specific IAM roles
Who manages: Development teams
Frequency of changes: Several times a day
State: Each service has its CloudFormation stack (in case of Serverless)
Contains:
- Lambda functions
- API Gateway configurations
- Lambda-specific IAM roles
- Custom domain mappings
- Environment variables
- Logs configuration

Principle: "Deploy often, fail fast, rollback easy"
Deploy: With each push to main (if changed files detected)
CI/CD: Automated, parallel, independent

LAYER 3: STATIC CONTENT (Direct S3 upload or CDN)
What: Frontend builds, static assets
Who manages: Frontend team
Frequency of changes: Several times a day
State: Stateless (direct file upload)
Contains:
- Built frontend files
- Static assets
- Public content

Principle: "Just upload it"
Deploy: npm run build && aws s3 sync
CI/CD: Automated, fast, no dependencies

===================================
INTERACTION BETWEEN LAYERS
===================================

APPROACH: Loose Coupling via Service Discovery

Terraform (Layer 1) exports outputs to SSM Parameter Store:
/infrastructure/{environment}/certificate-arn
/infrastructure/{environment}/hosted-zone-id
/infrastructure/{environment}/ecr-repository-url
/infrastructure/{environment}/domain-name

Serverless (Layer 2) reads these parameters via ${ssm:/path/to/parameter}

Advantages:
- No direct dependency between Terraform and Serverless states
- Can change implementation of any layer without affecting others
- Service discovery pattern - each service finds what it needs
- Versioning through parameter versions if needed

===================================
STATE MANAGEMENT STRATEGY
===================================

TERRAFORM STATE:
Location: S3 bucket with versioning enabled
Locking: DynamoDB table to prevent concurrent modifications
Structure:
  s3://your-state-bucket/
    infrastructure/
      terraform.tfstate          <- ONE state for entire foundation
    
Why one state for foundation:
- All shared resources have the same lifecycle
- Changes rarely and in controlled manner
- Dependencies between resources (certificate -> Route53)
- Easier to manage outputs

SERVERLESS STATE:
Location: CloudFormation stacks (managed by AWS)
Isolation: Each service = separate stack
Naming: {service-name}-{stage}
Examples:
  vabamorf-api-dev
  simplestore-api-dev
  audio-api-dev

Advantages:
- Automatic state management
- Built-in rollback
- Stack outputs for service discovery
- Parallel deploys without conflicts

===================================
CI/CD PRINCIPLES
===================================

1. CHANGE DETECTION
Deploy only what changed:
- Path filters to determine affected services
- Skip unchanged components
- Parallel execution where possible

Implementation:
  uses: dorny/paths-filter@v2
  filters:
    infrastructure: 'infrastructure/terraform/**'
    vabamorf: 'services/vabamorf-api/**'

2. DEPENDENCY ORDERING
Correct sequence:
  Foundation (Terraform) -> Applications (Serverless) -> Static Content

But: Applications can deploy in parallel with each other

Implementation via needs:
  deploy-services:
    needs: [deploy-infrastructure]

3. ENVIRONMENT PROMOTION
Dev -> Staging -> Production through same process:
- Same code
- Same tools
- Same commands
- Different variables

Implementation:
  STAGE=${{ github.ref == 'refs/heads/main' && 'prod' || 'dev' }}

4. FAIL FAST, ROLLBACK EASY
- Each service can rollback independently
- Infrastructure changes require approval
- Automatic rollback on failures

===================================
DOCKER IMAGES STRATEGY
===================================

BUILD ONCE, DEPLOY MANY:
1. Build Docker image in CI/CD
2. Tag with git commit SHA
3. Push to ECR
4. Serverless uses this image

Serverless automatically:
- Sees new image in ECR
- Creates new Lambda version
- Updates $LATEST alias
- Saves previous versions for rollback

Lifecycle:
- Terraform creates ECR repository (once)
- Terraform configures lifecycle policy (keep last N images)
- CI/CD pushes images with each deploy
- Serverless uses latest image

===================================
SCALING PATTERN
===================================

ADDING NEW SERVICE:
1. Check if new foundation resources needed (usually no)
2. Create directory services/new-service/
3. Add serverless.yml
4. Add path filter in CI/CD
5. Deploy

Not required:
- Change Terraform
- Touch other services
- Coordinate deploys

===================================
TROUBLESHOOTING PRINCIPLES
===================================

STATE LOCK ISSUES:
Root cause: Multiple processes trying to modify same Terraform state
Solution: Separate state by lifecycle boundaries

CIRCULAR DEPENDENCIES:
Root cause: Tight coupling between layers
Solution: Service discovery via SSM parameters

SLOW DEPLOYS:
Root cause: Sequential processing when parallel possible
Solution: Change detection + parallel execution

ROLLBACK DIFFICULTIES:
Root cause: Monolithic state with many resources
Solution: Independent stacks per service

===================================
MIGRATION STRATEGY
===================================

PHASES:

Phase 1: Create new structure (no changes to running infrastructure)
- Create new directory structure
- Write new Terraform configs (only shared resources)
- Write Serverless configs for each service
- DO NOT apply changes

Phase 2: Parallel run (both old and new exist)
- Deploy new infrastructure alongside old
- Use different resource names
- Verify it works
- Switch traffic gradually

Phase 3: Migrate state (careful!)
- Remove resources from old Terraform state
- Import into new structure
- Verify no changes in terraform plan

Phase 4: Cleanup old resources
- Delete old resources
- Delete old code
- Update documentation

Phase 5: Optimize
- Fine-tune CI/CD
- Adjust timeouts and retries
- Monitor and improve

===================================
KEY DECISIONS SUMMARY
===================================

DECISION: Terraform for infrastructure, Serverless for applications
REASON: Different lifecycles require different tools
ALTERNATIVE REJECTED: All-Terraform approach
WHY REJECTED: Would require complex state splitting, slower deploys, worse rollback story

DECISION: SSM Parameter Store for service discovery
REASON: Loose coupling between layers, no direct state dependencies
ALTERNATIVE REJECTED: Terraform outputs consumed directly
WHY REJECTED: Creates tight coupling, harder to evolve independently

DECISION: Change detection in CI/CD
REASON: Faster deploys, only deploy what changed
ALTERNATIVE REJECTED: Deploy everything always
WHY REJECTED: Wastes time, higher risk of breaking working services

DECISION: Parallel service deploys
REASON: Independent services should deploy independently
ALTERNATIVE REJECTED: Sequential deploys
WHY REJECTED: Slow, unnecessary serialization

===================================
SUCCESS METRICS
===================================

You know the architecture is correct when:
- Can deploy one service without touching others
- State lock conflicts disappeared
- Deploy time proportional to size of changes (not fixed overhead)
- Rollback of one service doesn't require rollback of others
- Adding new service requires minimal changes in infrastructure
- Developers can deploy without DevOps help
- Infrastructure changes happen rarely and in controlled manner

===================================
ANTI-PATTERNS TO AVOID
===================================

DON'T: Put frequently changing resources in same Terraform state as rarely changing
DON'T: Use Terraform for resources that deploy with every commit
DON'T: Share state between independent services
DON'T: Deploy everything when only one thing changed
DON'T: Use complex dependencies between Terraform modules when simple service discovery would work
DON'T: Put application configuration in Terraform (use application config management)
DON'T: Manual state file manipulation without backups

END OF DOCUMENT
===================================
