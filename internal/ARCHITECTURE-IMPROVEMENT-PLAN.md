# План улучшения архитектуры HAK

**Дата:** 2026-04-06
**Автор:** Kate (AI Agent)
**Статус:** Draft

## Контекст

По результатам архитектурного ревью hak-public выявлены проблемы, которые могут вызвать вопросы у независимых разработчиков. Этот план описывает шаги по их устранению.

---

## 1. State Management — внедрение Zustand

### Проблема
- 10+ взаимосвязанных hooks в `features/synthesis/hooks/synthesis/`
- `useSynthesis.ts` собирает их в мега-hook с 20+ методами
- Нет глобального state management

### Решение
Внедрить **Zustand** для глобального состояния synthesis.

### План
1. [ ] Создать `packages/frontend/src/stores/synthesisStore.ts`
2. [ ] Перенести состояние из `useSentenceState.ts` в store
3. [ ] Перенести audio state из `useAudioPlayer.ts` в store
4. [ ] Рефакторить `useSynthesisOrchestrator.ts` на использование store
5. [ ] Упростить `useSynthesis.ts` до thin wrapper над store
6. [ ] Удалить неиспользуемые hooks

### Вопросы
- **Q1:** Zustand или Jotai? Zustand проще, Jotai атомарнее.
- **Q2:** Переносить весь synthesis state или только shared части?

### Оценка
- **Сложность:** High
- **Время:** 2-3 дня
- **Риск:** Medium (много тестов завязано на текущую структуру)

## 2. Объединение .partN тестов

### Проблема
- 409 тестовых файлов, многие разбиты на `.part2.test.ts`, `.part3.test.ts`
- Признак того, что тесты слишком большие или код слишком сложный

### Решение
Объединить тесты где возможно, упростить код где нужно.

### План
1. [ ] Проанализировать все `.partN.test.ts` файлы
2. [ ] Определить причину разбиения (размер файла? время выполнения? coverage?)
3. [ ] Объединить тесты, которые можно объединить
4. [ ] Для остальных — упростить тестируемый код

### Вопросы
- **Q3:** Какой лимит на размер тестового файла? DevBox hook `source-size`?
- **Q4:** Есть ли тесты, которые намеренно разбиты (например, для параллельного запуска)?

### Оценка
- **Сложность:** Medium
- **Время:** 1-2 дня
- **Риск:** Low

## 3. Интернационализация (i18n)

### Текущее состояние
- Все UI на эстонском языке
- Строки централизованы в `config/ui-strings.ts` (подготовка к i18n)
- Нет react-i18next или аналога

### Решение
Пока **не внедрять** полноценный i18n. Проект ориентирован на эстонский рынок.

### План
1. [ ] Документировать текущий подход в ADR
2. [ ] Убедиться, что все строки в `ui-strings.ts`
3. [ ] Добавить комментарий в README о языковой политике

### Вопросы
- **Q5:** Нужен ли английский интерфейс в будущем? Или только эстонский?

### Оценка
- **Сложность:** Low
- **Время:** 0.5 дня
- **Риск:** None

## 4. Документация Merlin TTS

### Проблема
- Merlin TTS — устаревший синтезатор (2016)
- Независимый разработчик спросит "почему не Coqui/VITS?"

### Решение
Документировать причину выбора Merlin.

### План
1. [ ] Создать `docs/adr/009-merlin-tts-engine.md`
2. [ ] Объяснить: используем обученную модель от EKI
3. [ ] Описать ограничения и возможные альтернативы
4. [ ] Добавить ссылку в ARCHITECTURE.md

### Содержание ADR
- Merlin — единственный движок с обученной эстонской моделью от EKI
- Обучение новой модели не входит в scope проекта
- При появлении лучшей модели — можно заменить worker

### Оценка
- **Сложность:** Low
- **Время:** 0.5 дня
- **Риск:** None
