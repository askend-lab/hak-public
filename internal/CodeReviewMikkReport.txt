4. Code Style and readability, coding practices, quality issues

Validation topics: Code consistency, formatting, naming quality, clarity of logic.

Issue 1 (Low)
Problem: There are if statements in the project that do not include curly brackets.
Example: if (!response.ok) throw new Error(&quot;Synthesis request failed&quot;);
Suggestion: The problem with statements witohut brackets is that if you go back
and add a second statement without remembering to add the curly braces, the code
will break in unexpected and amusing ways.
Maintainability-wise, it’s always smarter to add curly brackets.
There also was a bug at Apple that was caused by not having curly brackets on if
statements, that caused issues with SSL/TLS:
https://www.imperialviolet.org/2014/02/22/applebug.html

Issue 2 (Medium)
Problem: There is a shared module that has shared functionality, but it is not always
used, nor correctly used.
For example, there is a shared method getCorsOrigin, but merlin-api and vabamorf-
api duplicate the same methods.

Suggestion: Use the shared functionality from shared module and remove
duplications.

Issue 3 (Low)
Problem: There are some usages of as unknown as in project source code (not
only tests). That is a bad idea as it bypasses compile-time type safety altogether.
Suggestion: Avoid double type assertions and define concrete, correctly typed
objects.

Issue 4 (Low)

Nested ternaries force the reader to mentally parse breaching structure and remove

Issue 5 (Low)
Problem: There are some occurences in project where arrays are used as index
keys.
&lt;div key={sIdx} className=&quot;specs-scenario__card&quot;&gt;
To optimize the rendering of React list components, a unique identifier (UID) is
required for each list item. This UID lets React identify the item throughout its
lifetime. Avoid array indexes since the order of the items may change, which will
cause keys to not match up between renders, recreating the DOM. It can negatively
impact performance and may cause issues with the component state.
Suggestion: Use a string or a number that uniquely identifies the list item. The key
must be unique among its siblings, not globally.
If the data comes from a database, database IDs are already unique and are the
best option. Otherwise, use a counter or a UUID generator.
More info: https://react.dev/learn/rendering-lists#rules-of-keys

13. Performance and scalability
Validation topics: Obvious performance issues, ineficcient queries (n+1 issues),
unnecessary complexity. Performance and scalability problems caused by
architectural decisions. Scalability of application to increase throughput and
performance.
Issue 1 (Medium)
Problem: DNN model is loaded from disk on every synthesis request.
dnn_model = pickle.load(open(nnets_file_name, &#39;rb&#39;))
pickle.load is called inside dnn_generation, which is invoked twice per request
(duration model + acoustic model).
Unpickling a neural network from disk on every single TTS request is expensive.
That causes latency per every request.
Suggestion: Load the models once at process startup and reuse them across
requests, instead of unpickling from disk per request.

Issue 2 (Medium)
Problem: SQS Worker polls only 1 message per cycle
while not _shutdown_requested:
try:
resp = sqs_client.receive_message(
QueueUrl=config.queue_url,
MaxNumberOfMessages=1,
WaitTimeSeconds=POLL_WAIT_SECONDS,

VisibilityTimeout=VISIBILITY_TIMEOUT,
MaxNumberOfMessages=1 means each 20-second long-poll cycle retrieves at most
one message. Under load (queue backlog), the worker processes messages one at
a time with significant overhead per cycle. SQS supports up to 10 messages per
receive call.
The impact is that throughput will be bottlenecked under load.
Suggestion: Validate if this is relevant in project case. If yes, increase batch retrieval
and process messages in batches to reduce polling overhead and improve
throughput. I would say that easy change is setting MaxNumberOfMessages to 10.
